# 렌더링 성능 비교: Canvas2D vs Metal

## 재현 방법

두 벤치마크 도구는 dev/debug 전용이며 프로덕션 빌드에 포함되지 않는다.

### 웹 (Canvas2D)

```bash
bun run dev
# http://localhost:5173/bench 접속
# "Run Benchmark" 클릭 → "Copy as Markdown" 클릭
```

- **소스**: `src/routes/bench/+page.svelte`
- **가드**: `src/routes/bench/+page.ts`에서 프로덕션 환경 시 404 반환 (`$app/environment.dev` 체크)
- **측정 대상**: `src/lib/canvas/renderer.ts`의 `renderPixelCanvas()`

### Apple (Metal)

1. Xcode에서 **Debug** 스킴으로 빌드
2. 툴바의 **"Benchmark"** 버튼 탭 → 시트 열림
3. **"Run Benchmark"** 탭 → **"Copy as Markdown"** 탭

- **소스**: `apple/Dotorixel/Benchmark/RenderBenchmarkView.swift` (`#if DEBUG`)
- **렌더러 메서드**: `PixelGridRenderer.benchmarkDraw(in:)` (`#if DEBUG`)
- **진입점**: `ContentView.swift` 툴바 버튼 (`#if DEBUG`)

## 측정 방법론

두 벤치마크 모두 동일한 접근 방식 사용: 강제 반복 렌더링으로 프레임당 소요 시간 측정.

- **워밍업**: 10프레임 (JIT 컴파일, GPU 파이프라인 안정화)
- **측정**: 100프레임
- **통계**: median, P95, 이론적 최대 FPS (1000 / median_ms)
- **뷰포트**: 512×512 픽셀
- **그리드**: 활성화 (체커보드 + 픽셀 + 그리드 3단계 모두 활성화하는 최악 조건)
- **캔버스 데이터**: 모든 픽셀을 반투명 색상으로 채움 (RGBA 128, 64, 192, 180)

### 테스트 매트릭스

캔버스 크기 5종 × 줌 레벨 3종 = 15개 구성:

| 캔버스 크기 | 줌 레벨 |
|------------|---------|
| 8×8, 16×16, 32×32, 64×64, 128×128 | 1×, 8×, 16× |

### Effective Pixel Size

`effectivePixelSize = round(pixelSize × zoom)`, `pixelSize = floor(512 / canvasSize)`.

| 캔버스 | pixelSize | @1× | @8× | @16× |
|--------|-----------|-----|-----|------|
| 8×8 | 64 | 64 | 512 | 1024 |
| 16×16 | 32 | 32 | 256 | 512 |
| 32×32 | 16 | 16 | 128 | 256 |
| 64×64 | 8 | 8 | 64 | 128 |
| 128×128 | 4 | 4 | 32 | 64 |

### 플랫폼 차이

- **Canvas2D** (웹): 동기 API. `renderPixelCanvas()` 호출 전후 `performance.now()`로 측정. 매 프레임 `OffscreenCanvas`를 새로 할당하여 픽셀 합성.
- **Metal** (Apple 네이티브): 비동기 GPU 파이프라인. CPU 타이밍은 `commandBuffer.commit()` + `waitUntilCompleted()` 포함. GPU 타이밍은 `commandBuffer.gpuStartTime`/`gpuEndTime` 사용. 단일 fullscreen quad — 모든 렌더링 로직이 fragment shader에서 실행.

## 분석

### 스케일링 특성

**Canvas2D**는 캔버스 크기(픽셀 수)에 비례하여 느려진다:

| 캔버스 | 1× (ms) | 8× (ms) | 16× (ms) |
|--------|---------|---------|----------|
| 8×8 | 0.100 | 0.000 | 0.000 |
| 32×32 | 0.400 | 0.200 | 0.200 |
| 128×128 | 1.600 | 3.300 | 3.300 |

렌더러가 모든 `width × height` 픽셀을 JavaScript 루프로 순회하며 `fillRect()`를 호출한다. 128×128에서는 렌더 단계(체커보드, 픽셀, 그리드)마다 16,384번 반복. 고배율에서는 2×2 서브 체커보드 모드(effective pixel size ≥ 8px에서 활성화)가 `fillRect()` 호출을 4배로 늘린다: 128×128 × 4 = 65,536 draw call(체커보드만).

**Metal**은 캔버스 크기와 줌에 무관하게 GPU 시간이 거의 일정하다:

| 캔버스 | 1× (ms) | 8× (ms) | 16× (ms) |
|--------|---------|---------|----------|
| 8×8 | 0.063 | 0.063 | 0.064 |
| 32×32 | 0.071 | 0.070 | 0.063 |
| 128×128 | 0.048 | 0.063 | 0.063 |

Fragment shader가 항상 동일한 뷰포트 픽셀(512×512 drawable)을 처리한다. 캔버스 크기는 텍스처 차원에만 영향을 미치고, 줌은 uniform 값일 뿐 — GPU 작업량이 변하지 않는다.

### 렌더러 간 비교

| 캔버스 | 줌 | Canvas2D (ms) | Metal CPU (ms) | Metal GPU (ms) | 비율 (Canvas2D / Metal CPU) |
|--------|------|--------------|----------------|----------------|------------------------------|
| 8×8 | 1× | 0.100 | 0.249 | 0.063 | 0.4× (Canvas2D가 더 빠름) |
| 32×32 | 1× | 0.400 | 0.326 | 0.071 | 1.2× |
| 64×64 | 1× | 1.400 | 0.231 | 0.063 | 6.1× |
| 128×128 | 1× | 1.600 | 0.207 | 0.048 | 7.7× |
| 128×128 | 8× | 3.300 | 0.226 | 0.063 | 14.6× |
| 128×128 | 16× | 3.300 | 0.226 | 0.063 | 14.6× |

작은 캔버스(8×8)에서는 Canvas2D가 오히려 더 빠르다. Metal은 command buffer 생성/제출에 ~0.2ms의 고정 오버헤드가 있어 극히 적은 작업량에서는 이 비용이 지배적이다. 교차점은 32×32 부근.

큰 캔버스에서는 격차가 급격히 벌어진다: 128×128 고배율에서 Metal이 15배 더 빠르다. Canvas2D의 O(n²) CPU 루프 vs Metal의 O(1) GPU fragment shader라는 아키텍처 차이가 그대로 드러난다.

### 절대 성능

두 렌더러 모두 테스트한 모든 구성에서 60fps 기준(16.7ms) 이내:

- **Canvas2D 최악**: 3.3ms (128×128 @ 16×) = 프레임 예산의 20%
- **Metal 최악**: 0.33ms (32×32 @ 1×) = 프레임 예산의 2%

현재 MAX_DIMENSION 128×128에서 실시간 인터랙션은 양쪽 모두 문제없다.

## 부록: 실제 픽셀 아트 캔버스 크기

벤치마크 결과에 맥락을 제공하기 위해, 실제 픽셀 아트 게임에서 사용되는 캔버스 크기를 정리한다.

### 캐릭터 스프라이트

| 디테일 수준 | 크기 | 참고 게임 |
|---|---|---|
| 저해상도 (레트로) | 16×16, 16×32 | Stardew Valley, Celeste, Undertale |
| 중간 | 32×32, 32×48 | Shovel Knight, RPG Maker XP, HD-2D 게임 (Octopath Traveler, DQ III) |
| 고해상도 (하이비트) | 48×48, 64×64 | Hyper Light Drifter, Dead Cells |
| 대형 / 보스 | 80–150px, 최대 200–300px | Blasphemous (플레이어 ~80–100px), Children of Morta (최대 ~290px) |

HD-2D 게임(Octopath Traveler, Triangle Strategy, Dragon Quest III HD-2D)은 의도적으로 작은 스프라이트(~32×48 ~ 48×64)를 사용한다. "고화질"은 스프라이트가 아닌 3D 환경과 후처리 효과에서 온다.

### 타일

| 크기 | 사용처 |
|---|---|
| 16×16 | 가장 보편적: Stardew Valley, Shovel Knight, RPG Maker 2000–VX Ace |
| 32×32 | RPG Maker XP/VX/VX Ace 기본 타일 |
| 48×48 | RPG Maker MV/MZ 기본 타일 |

타일셋 시트(개별 타일을 조합)는 256×256, 512×512, 1024×1024에 이를 수 있다.

### 배경 및 풀스크린

| 유형 | 크기 | 참고 |
|---|---|---|
| 레트로 게임 화면 | 320×180, 320×240 | Celeste (320×180, 6× 스케일로 1080p) |
| 하이비트 게임 화면 | 480×270, 640×360 | Dead Cells (~480×270), Blasphemous / Owlboy / Sea of Stars (640×360) |
| 독립 일러스트 | 128×128, 256×256, 512×512 | 포트폴리오, SNS |
| 스프라이트 시트 | 1024×1024, 2048×2048 | 게임 엔진 에셋 파이프라인 |

640×360 해상도는 사실상의 "하이비트" 표준이며, 1080p (3×)와 4K (6×)로 정수배 스케일링된다.

### 캔버스 크기 제한별 커버 범위

| 제한 | 커버 범위 |
|---|---|
| **128×128** (현재) | 레트로/인디 스프라이트, 개별 타일, HD-2D 스프라이트, 아이콘, 소규모 일러스트 |
| **256×256** | 대형 보스 스프라이트 (Blasphemous급), 소규모 타일셋 시트, 중간 크기 일러스트 |
| **640×360** | 하이비트 게임 풀스크린, 패럴랙스 배경 |
| **1024+** | 스프라이트 시트 조합, 대형 타일셋 — 전문 에셋 파이프라인 영역 |

### 결론

1. **현재 규모에서는 양쪽 렌더러 모두 최적화가 불필요하다.** 최악 조건에서도 수백~수천 FPS를 달성하며, 60fps 목표를 쉽게 충족한다.
2. **현재 128×128 제한은 대부분의 개별 에셋 용도를 커버한다** — 레트로/인디 스프라이트, 개별 타일, HD-2D 스프라이트, 소규모 일러스트. 대형 보스 스프라이트(Blasphemous급)와 풀스크린 배경은 부족하다.
3. **256×256으로 확장 시** 대형 스프라이트와 소규모 일러스트를 커버할 수 있다. Canvas2D는 ~13ms로 추정(16.7ms 프레임 예산 근접)되어 뷰포트 클리핑 등 최적화가 권장된다. Metal은 1ms 미만을 유지한다.
4. **640×360+ 확장 시** (하이비트 풀스크린 배경) Canvas2D 최적화(뷰포트 클리핑) 또는 WebGL2 전환이 필요하다. Metal의 상수 시간 아키텍처는 변경 없이 대응 가능하다.
5. **Canvas2D의 매 프레임 `OffscreenCanvas` 할당**은 현재 크기에서 병목이 아니지만, 큰 캔버스에서는 누적된다.
6. **Metal의 고정 오버헤드(~0.2ms)**는 실제 렌더링이 아닌 command buffer 관리 비용이다. 픽셀 아트의 on-demand 렌더링 모델(변경 시에만 재그리기)에서는 무관하다.
