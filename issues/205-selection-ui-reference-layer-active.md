## Problem Statement

### Steps to reproduce

1. With a Pixel Layer active, make an area selection (create a Marquee).
2. Activate the Reference Layer.
3. The SelectionActionBar (selection toolbar) stays visible, and its Deselect
   button cannot clear it — the selection UI is stuck with no way to dismiss it.

### Root cause (code-traced)

The Marquee is document-scoped, so it survives the active-layer switch. Two
independent gaps then combine:

1. **Visibility ignores the active-layer kind.** `SelectionActionBar` renders
   whenever a projected Marquee exists; it does not consult `activeLayerKind`.
   So it shows even when a Reference Layer is active — where its actions (copy /
   cut / flip / rotate / delete) are all no-ops.
2. **The reference placement overlay swallows the Deselect click.** Deselect is
   correctly wired (`onClearMarqueeOrFloating` → `clearMarqueeOrFloating` →
   `clearMarquee` → `set_marquee(null)`), and none of that path is gated on
   layer kind — so at the model level it *would* clear. But
   `ReferenceLayerPlacementOverlay` (rendered only while a Reference Layer is
   active, `pointer-events: auto`, higher stacking than the toolbar) covers the
   reference's on-screen footprint and intercepts the click before it reaches
   the Deselect button.

The bar behaves correctly whenever no Reference Layer is active — the reference
placement overlay is the only differentiator.

## Agent Brief

**Category:** bug

**Summary:** Suppress the on-canvas selection UI while a Reference Layer is the
active layer, so it can neither get stuck nor become unclickable.

**Current behavior:**
A Marquee made on a Pixel Layer persists (document-scoped) when the user
activates the Reference Layer. The SelectionActionBar keeps showing, and because
the reference placement overlay renders above it and captures pointer events over
the reference footprint, the Deselect button can't be clicked — the selection UI
is stuck with no way to dismiss it while the Reference Layer is active.

**Desired behavior:**
While the active layer is a Reference Layer, no selection UI is shown — neither
the SelectionActionBar nor the on-canvas Marquee outline. The Marquee region
itself is preserved in the Document (not cleared), so switching back to a Pixel
Layer restores both the outline and the toolbar exactly as before. Selection
behavior on Pixel Layers is unchanged.

**Key interfaces / concepts:**

- `SelectionActionBar` visibility should be conditioned on the active layer being
  a Pixel Layer. The layer projection already exposes `activeLayerKind`, and the
  existing `nudgeMarquee` and reference-sampling paths already early-return on
  `activeLayerKind === 'reference'` — follow that established precedent.
- The on-canvas Marquee outline rendering should be gated the same way.
- The Marquee (`document.marquee`) must **not** be cleared on the layer switch —
  it stays document-scoped and is only hidden from view.

**Acceptance criteria:**

- [ ] With a Marquee present, activating the Reference Layer hides the
      SelectionActionBar.
- [ ] The on-canvas Marquee outline is likewise not shown while a Reference Layer
      is active.
- [ ] The Marquee is preserved: switching back to a Pixel Layer restores the
      outline and the SelectionActionBar unchanged.
- [ ] On a Pixel Layer, the SelectionActionBar and its Deselect action work
      exactly as before (regression).
- [ ] A regression test covers "selection UI hidden while a Reference Layer is
      active, Marquee preserved across the switch."

**Out of scope:**

- Changing the Marquee's document-scoped persistence model.
- Restacking / z-index changes to the reference placement overlay — fix via
  visibility gating, not by re-stacking.
- The Reference Footprint / reference placement geometry (issue 204 / PR #295).
- Any Reference Window (floating) behavior — this is about the in-document
  Reference Layer underlay only.

## Results

| File | Description |
|------|-------------|
| `src/lib/canvas/PixelCanvasView.svelte` | `displayedMarquee` derived nulls the marquee fed to `SelectionOverlay` + `SelectionActionBar` while a Reference Layer is active — display-only gating; the document Marquee is untouched |
| `src/lib/canvas/PixelCanvasView.svelte.test.ts` | New behavior tests: action bar hidden, outline hidden, selection UI restored unchanged after switching back; replaces the 138-era "geometry unchanged while Reference is active" test |
| `crates/core/src/document.rs` | Regression pin: `set_active_layer` preserves the Marquee across a Reference Layer switch and back |

### Key Decisions

- Gated at `PixelCanvasView`, the single composition point of both selection
  components — covers both editor layout variants without changing child
  interfaces or the model.
- **Supersedes issue 138's "visible-but-inert Marquee" UX.** 138 intentionally
  kept the marching ants visible while a Reference Layer is active; in practice
  the SelectionActionBar showed only no-op actions and the placement overlay
  swallowed the Deselect click (this bug). 205 reverses that decision: the
  selection UI is hidden, the Marquee is preserved. The 138 visual test was
  replaced by a restore-unchanged test that keeps its geometry-preservation
  intent.
- No gating on `floatingSelectionOffset`: `setActiveLayer` commits a pending
  Floating Selection before switching, so it can never coexist with a
  Reference-active document — no redundant guard added.
