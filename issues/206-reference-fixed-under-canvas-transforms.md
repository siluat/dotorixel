## Problem Statement

Whole-canvas Flip and Rotate treat the Reference Layer inconsistently:

| | Whole-canvas **Flip** | Whole-canvas **Rotate** (no Marquee) |
|---|---|---|
| Reference Layer | **left unchanged** | **placement rotated** (quarter-turn advanced + repositioned) |
| Pixel Layers | active layer / active frame only | all Pixel Layers, all frames |
| Canvas size | unchanged | width/height swap |

So rotating the canvas turns the reference image with it, while flipping the
canvas leaves the reference untouched — the same "whole-canvas" mental model
produces different results.

### Why it is this way (data model)

`ReferencePlacement` carries a quarter-turn `rotation` field but no mirror/flip
concept, so whole-canvas Rotate can apply to the reference while Flip has no
mechanism to. Reference rotation is produced **only** by whole-canvas Rotate
(there is no independent "rotate the reference" control) and is persisted /
restored on load.

### Decision: exclude the Reference Layer from whole-canvas transforms (fixed overlay)

The Reference Layer is a fixed tracing overlay, not part of the transformable
artwork. Whole-canvas Rotate should stop transforming it, so that **neither**
Flip nor Rotate touches the reference. This matches the convention of the
reference tool this feature is modeled on (see Notes) and needs no data-model
change. (The rejected alternative — making Flip also mirror the reference — is
covered under Out of scope.)

## Agent Brief

**Category:** bug

**Summary:** Whole-canvas Rotate must leave the Reference Layer's placement
unchanged, matching Flip — the Reference Layer is a fixed overlay excluded from
whole-canvas transforms.

**Current behavior:**
With no Marquee, rotating the whole canvas rotates every Pixel Layer, swaps the
canvas dimensions, **and** remaps the Reference Layer's placement (advances its
quarter-turn and repositions it so it turns with the canvas). Flipping the
canvas, by contrast, never affects the Reference Layer.

**Desired behavior:**
Whole-canvas Rotate leaves the Reference Layer's placement (position, scale,
quarter-turn) unchanged — only the Pixel Layers rotate and the canvas
width/height swap. The reference stays fixed in document space while the canvas
reshapes around it. Flip already leaves the reference unchanged, so after this
change neither whole-canvas transform touches the reference.

Backward compatibility: a Document that already contains a rotated reference
(quarter-turn ≠ 0, produced/persisted before this change) must still render and
sample that reference correctly. This change only stops **producing** new
reference rotations; it does not remove the ability to display an existing one.

**Key interfaces / concepts:**

- The Canvas Transform rotate op (the explicit whole-document rotate that
  issue 209 establishes) should skip Reference Layers entirely: it transforms
  only Pixel Layers and swaps the canvas dimensions. It must not advance the
  reference's quarter-turn or reposition it. The now-unused reference-rotation
  remap helper it calls today should be removed.
- The reference rotation *rendering* machinery must remain — the placement's
  quarter-turn field, the rotation-aware Reference Footprint, the sampling
  inverse map, and quarter-turn persistence/hydration — so already-saved rotated
  references still display and sample correctly.

**Acceptance criteria:**

- [ ] Rotating the whole canvas (no Marquee) with a Reference Layer present
      leaves the reference's position, scale, and quarter-turn unchanged; only
      the Pixel Layers rotate and the canvas width/height swap.
- [ ] Flipping the canvas continues to leave the Reference Layer unchanged (no
      regression).
- [ ] A Reference Layer that already has a non-zero quarter-turn still renders
      and samples correctly (backward compatibility with saved documents).
- [ ] The existing whole-canvas rotate tests that asserted the reference is
      remapped / its quarter-turn advanced are updated to assert the reference is
      unchanged across the rotate.
- [ ] `docs/platform-status.md` and any domain docs no longer state that the
      reference "turns with a whole-document rotate."

**Out of scope:**

- Adding a mirror/flip capability to the reference placement so Flip could mirror
  the reference (the rejected "reference follows the artwork" alternative).
- Migrating already-saved rotated references to un-rotated — keep rendering them
  as saved; do not rewrite persisted data.
- The separate scope difference between Flip (active Pixel Layer / active frame)
  and Rotate (all Pixel Layers / all frames) — addressed by the parent PRD 207's
  other sub-issues (208 / 209 / 210), not this reference-consistency slice.
- The rotation-aware Reference Footprint itself (issue 204) — it stays; it is
  still needed to render already-saved rotated references.

## Notes

- Re-parented under PRD 207 (tiered transforms) during its breakdown — this
  issue's reference-exclusion decision became part of the Canvas Transform tier
  definition. The agent brief above remains the authoritative spec for the
  slice.
- **Decision rationale (Option B over Option A).** Chose "reference is a fixed
  overlay, excluded from both transforms" over "reference follows the artwork
  through both." Rationale: Aseprite — the reference-layer implementation this
  feature is modeled on — treats reference layers as **Move-tool-only overlays
  that cannot be rotated or flipped** ("This is not possible yet," per its
  developer), i.e. a fixed overlay outside canvas transforms; and Option B needs
  no `ReferencePlacement` data-model change (Option A would require a persisted
  mirror field threaded through render / sampling / persistence / both shells).
  Research:
  [Aseprite — can you rotate the reference layer?](https://community.aseprite.org/t/can-you-rotate-the-reference-layer/705),

  [Pixquare — Canvas docs](https://docs.pixquare.art/settings/canvas).
