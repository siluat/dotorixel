## Problem

`InputMTKView` forwards touch events without touch identity
(`isMultipleTouchEnabled` is on): `touchesBegan` fires again for a second
finger and replaces the active stroke session (`EditorState.beginStroke`
cancels the previous one first), and `touchesEnded` fires when **any** finger
lifts — so lifting the first finger prematurely ends the second finger's
stroke, and the still-down finger's further movement is ignored.

History stays consistent through this sequence — each session's Stroke
Baseline resolves correctly (see issue 243) — but the stroke UX is wrong:
pre-existing input-routing behavior, not a 243 regression.

## Directions to evaluate

- Associate the active stroke with its originating `UITouch` and ignore
  begin/move/end from other touches.
- Or suppress drawing while a second touch is down (web parity: the web
  interaction machine ends the stroke and waits for pinch when a second
  touch arrives).
- Coordinate with the planned Apple Pencil palm-rejection work — both decide
  which touches may draw.

## Agent Brief

**Category:** bug
**Summary:** A second concurrent touch must stop drawing and yield to
viewport gestures (web parity), instead of restarting the stroke and letting
any finger's lift end it.

**Current behavior:**
The Metal input view forwards touch events without touch identity. A second
finger's `touchesBegan` cancels the active stroke and starts a new one at the
second finger's location; `touchesEnded` for **any** finger ends the active
stroke, so lifting the first finger prematurely ends the second finger's
stroke and the still-down finger's further movement is ignored. `touchesMoved`
reads an arbitrary touch from the set, so two moving fingers can interleave
both touches' coordinates into one stroke.

**Desired behavior (web parity):**
The web shell's canvas interaction machine is the reference: when a second
touch arrives during a touch draw, drawing stops and the machine waits for
the pinch/pan gesture; drawing does not resume until all touches lift.
Mapped onto the Apple shell:

- A stroke session is associated with its originating touch. Only that
  touch's begin/move/end/cancel events drive the session; other touches'
  events never feed it.
- A second concurrent touch never starts a new stroke session.
- When a second touch begins while a finger stroke is active:
  - If the stroke has been dragged (received samples beyond its initial
    touch-down sample), it ends through the normal commit path — drawn
    pixels stay, its single history entry resolves as usual.
  - If the stroke still holds only its initial touch-down sample, it
    resolves with pixels and history untouched (matching the web shell,
    where a quick second finger discards the not-yet-committed tap, so a
    pinch start never leaves a stray dot).
- While two touches are down, the existing pinch-zoom and two-finger-pan
  recognizers operate unchanged.
- After a multi-touch episode, no stroke begins until **all** touches have
  lifted; a fresh single touch then draws normally.

**Key interfaces:**
- The `UIView` touch handlers (`touchesBegan/Moved/Ended/Cancelled`) on the
  Metal input view — currently read `touches.first` and forward without
  identity; this is where originating-touch association belongs.
- `CanvasInputDelegate` — the platform-input abstraction; extend its
  contract only if the routing decision cannot be made at the view layer.
- `EditorState.beginStroke(at:button:)` — currently force-cancels any
  active stroke when a begin arrives mid-stroke. With routing fixed, a
  mid-stroke begin should no longer occur from multi-touch; keep its
  defensive behavior but do not rely on it for this feature.
- `EditorState.endStroke()` / `cancelStroke()` — the commit and discard
  paths the second-touch rule resolves through; both already resolve the
  Edit Baseline (issue 243 semantics) and must keep doing so.

**Acceptance criteria:**
- [ ] With a stroke active, a second touch down never starts a new stroke
      at the second touch's location.
- [ ] A dragged stroke interrupted by a second touch commits exactly what
      was drawn before the second touch, as one history entry.
- [ ] A stroke holding only its initial sample when a second touch arrives
      leaves canvas pixels and undo history unchanged.
- [ ] Only the originating touch's moves feed the active stroke — a second
      touch's coordinates never appear in the stroke's samples.
- [ ] Lifting a touch other than the originating one does not end an
      active stroke (relevant while the second-touch rule is resolving, and
      for any future state where a non-drawing touch is down).
- [ ] Pinch-zoom and two-finger pan continue to work while two fingers
      are down.
- [ ] After a two-finger gesture, movement of a still-down finger draws
      nothing; once all touches lift, the next single touch draws normally.
- [ ] Behavior is pinned by Swift Testing tests at whatever seam the
      implementation makes testable (stroke-lifecycle outcomes — pixels,
      history, session state — not event-forwarding call sequences).

**Out of scope:**
- Apple Pencil palm rejection (pencil-vs-finger priority, hover preview) —
  this issue only establishes the single-drawing-touch gate that the
  palm-rejection work will later refine.
- Porting the web shell's deferred tap commit (`pendingCoords`) and
  long-press color-sampling arbitration as a general mechanism — only the
  narrow "initial-sample-only stroke resolves clean on second touch"
  outcome is required here.
- macOS mouse/trackpad paths and the web shell — unchanged.
- Reworking `EditorState.beginStroke`'s defensive cancel-on-begin.

### Key Decisions

- **Deferred Begin instead of the cancel path** for the "initial-sample-only stroke
  resolves clean" criterion: `FreehandStrokeSession.cancel()` deliberately keeps
  painted pixels (interrupted strokes preserve work), so cancelling could never
  un-paint the touch-down dot. Deferring the begin — the web's own mechanism
  (`pendingCoords`) — means the dot never lands, without touching any session
  or History semantics.
- **Shell-native placement** (no Rust core involvement), per the Core Placement
  criteria: the logic is inseparable from UIKit touch identity and gesture-
  recognizer interplay (platform divergence), sits on the per-event input hot
  path (binding friction), and the web shell already owns an equivalent machine
  (no cross-platform sharing to gain).
- **Gesture Signal counts direct touches only** (web parity: only `touch`
  pointers enter the two-pointer check). Bonus beyond the brief's minimum: a
  single resting finger neither replaces, feeds, nor ends a pencil stroke —
  a pre-palm-rejection tolerance that the palm-rejection work will extend.
- Router is a generic pure `struct` (`TouchStrokeRouter<TouchID: Hashable>`),
  compiled into both platforms (CoreGraphics only) — the testable seam the
  agent brief asked for, and the seam palm rejection will refine.
