## Problem

The drawing-tool effect `commitFloatingSelection` is declared in the `ToolEffect`
union and handled in the session's effect applier, but **no tool ever emits it**.
It is a dead branch.

It is worth more than a routine dead-code cleanup because of *what* the branch
does. The handler commits the floating selection directly, whereas the
supported commit path goes through the lifecycle's `commitIfPending`, which
refuses while a stroke is in progress (`getIsDrawing`). So if some future tool
started emitting this effect, it would commit a floating selection mid-stroke —
bypassing a seal the rest of the system relies on, including the precondition
that undo/redo and baseline capture are never re-entered while a stroke is live
(debug-asserted in the core History ring).

## Solution

Confirm the effect is genuinely unused, then either delete it (union member and
handler branch) or, if a caller is actually intended, route it through the
guarded commit path so the `isDrawing` seal holds.

Deleting is the likely answer: the effect union is the tools' vocabulary, and
carrying a member no tool speaks invites someone to reach for the unguarded
path precisely because it looks supported.
