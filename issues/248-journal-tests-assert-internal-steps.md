## Problem

`DocumentChangeJournal`'s tests record collaborator calls into an `events`
array and assert the exact sequence. Some of those entries are genuine outcomes
the journal owns — `render`, `dirty`, `reclamp`, `sync`. Others name *which
History API the journal happened to call* (`begin-edit`, `end-edit`, and
`snapshot` before it), which is implementation, not behavior.

`testing.md` states the warning sign directly:

> If refactoring the internals breaks a test without changing behavior, the
> test was too tightly coupled.

Issue 244 realised it. Routing commands through the Edit Baseline changed no
externally observable behavior for a command that *does* change the document —
same undo entry, same follow-up effects, same `commit()` return — yet **28
tests broke**, purely because the journal now calls `begin_edit`/`end_edit`
where it used to call `push_document`.

## Solution

Assert outcomes against the real `DocumentHistory` rather than the sequence of
calls into a fake:

- History outcomes → `journal.canUndo` / `canRedo`, and undo actually restoring
  the prior state. 244 added three tests in this style (the blank-layer clear,
  its redo future, the blank square canvas rotate); they exercise the real
  comparison and would survive this kind of refactor.
- Keep the `events` array for the side effects the journal genuinely owns
  (render, dirty, reclamp, sync) — those are outcomes, not internal steps.

## Notes

- **One coupling is worth keeping, deliberately**: the baseline must open
  *before* the apply, or the undo entry captures post-edit state and restores
  the wrong thing. That ordering is a real contract and deserves one dedicated
  test — but as one test, not as ambient coupling in ~40.
- The fake history is still the right tool for wiring tests, where the point is
  that the journal reacts correctly to a verdict it is handed (244 added
  `endEditCommits` to control it). The problem is scope, not the fake's
  existence.
- Scope check during triage: this is a test-only refactor with no production
  change, so its value is the next refactor it stops taxing. Weigh it against
  simply leaving the tests alone — the churn 244 paid is now already paid.

## Agent Brief

> *This was generated by AI during triage.*

**Category:** enhancement
**Summary:** Refactor `DocumentChangeJournal` tests to assert history
*outcomes* (undo/redo availability and state restoration) instead of the
sequence of History API calls recorded into a fake, keeping exactly one
dedicated test for the baseline-before-apply ordering contract.

**Current behavior:**
The journal test suite drives a fake `DocumentHistory` that records every
collaborator call into a shared `events` array, and many tests assert the
exact sequence — including `begin-edit`/`end-edit` entries that name *which
History API the journal called*. Issue 244 rerouted commands through the Edit
Baseline with no externally observable behavior change and broke 28 such
tests. A minority of tests (added by 244) already use the target style: the
real history from the WASM backend, asserting `journal.canUndo`/`canRedo` and
that undo restores the prior document state.

**Desired behavior:**
- Tests whose point is a history effect assert outcomes: `journal.canUndo` /
  `journal.canRedo`, `commit()`'s `changed` verdict, and — where the scenario
  warrants it — that `undo()`/`redo()` actually restores the expected document
  state. Prefer the real history factory (as in 244's no-op tests) when the
  scenario runs against real documents.
- `events` assertions keep only side effects the journal genuinely owns as
  outcomes — render invalidation, dirty marking, viewport reclamp, metrics
  sync, document replacement, and collaborator effects that *are* the
  intent's result (e.g. adding a layer). No assertion outside the dedicated
  ordering test names a History API call.
- **One dedicated test** pins the deliberate coupling: the undo baseline must
  open *before* the intent applies to the document (otherwise the undo entry
  captures post-edit state). This is a real contract and keeps its
  call-sequence assertion — but as one test, not ambient coupling.
- The fake history remains for wiring tests where the test dictates the
  commit verdict (the `endEditCommits`-style control) — the problem is the
  scope of sequence assertions, not the fake's existence.

**Key interfaces:**
- `DocumentChangeJournal` public surface — `commit()` and its `changed`
  result, `undo()`/`redo()`, `canUndo`/`canRedo` — is the assertion surface;
  it does not change.
- The `DocumentHistory` contract (`begin_edit`/`end_edit`/`undo`/`redo`/
  `clear`/`can_undo`/`can_redo`) — calls into it are implementation detail
  except for the one ordering-contract test.
- The real history factory and document helpers exported by the WASM backend
  (`createDocumentHistory`, `singleLayerDocument`, `clearActiveLayerPixels`)
  — the pattern 244's outcome-style tests already use.

**Acceptance criteria:**
- [ ] Exactly one test asserts the relative order of a History API call
      (baseline opens before the intent applies); no other assertion names
      `begin-edit`/`end-edit` or an equivalent History call marker.
- [ ] Every behavior previously specified via a History call sequence is
      still specified, now via `canUndo`/`canRedo`, the `changed` verdict,
      and/or actual undo/redo state restoration — no test simply deleted
      without its behavior re-covered.
- [ ] `events` assertions cover only journal-owned outcomes (render, dirty,
      reclamp, sync, replacement, collaborator effects that are the intent's
      result).
- [ ] The full web test suite passes with zero production-code changes.

**Out of scope:**
- Any change to production code (`DocumentChangeJournal`, history, WASM
  backend).
- Removing the fake history entirely — it stays for verdict-controlled
  wiring tests.
- Other test files, including the Rust core's history tests.
