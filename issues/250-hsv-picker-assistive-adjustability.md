## Problem Statement

The HSV picker is drag-only on both shells. Assistive technologies can read
the current values but cannot change them:

- **Web** (`src/lib/color-picker/HsvPicker.svelte`): the SV area and hue strip
  expose `role="slider"`, `tabindex`, and ARIA values, but there are no
  keyboard handlers — Tab reaches the controls, arrow keys do nothing.
- **Apple** (`apple/Dotorixel/Views/HsvPickerView.swift`): the SV square and
  hue strip expose `accessibilityLabel`/`accessibilityValue`, but no
  adjustable or custom actions — VoiceOver users cannot operate them.

Flagged by coderabbitai and cubic-dev-ai on PR #330 (issue 238); deferred
there because the gap is cross-shell and needs shared interaction design.

## Solution (sketch — needs design decision)

The SV square is two-axis, while adjustable actions/arrow keys are one-axis.
Decide one mapping and mirror it on both shells:

- **Option A — split axes**: expose saturation and brightness as separate
  adjustable elements (Apple: two `accessibilityAdjustableAction` elements;
  web: arrow Left/Right = saturation, Up/Down = brightness on the SV area).
  Hue strip is a plain one-axis slider on both.
- **Option B — custom actions** (Apple): named increment/decrement actions
  per dimension via `accessibilityCustomAction`; web still needs Option A's
  keyboard handling.

Whichever mapping is chosen, edits must reuse the existing model paths
(`HsvPickerModel` on Apple, `hsv` state + `emitColor` on web) so behavior
stays in sync with pointer edits.

## Acceptance criteria

- Web: SV area and hue strip are fully keyboard-operable with visible focus,
  ARIA values updating live.
- Apple: VoiceOver can adjust saturation, brightness, and hue; announced
  values update live.
- Foreground color updates identically to pointer edits on both shells.
