## Problem Statement

The HSV picker is drag-only on both shells. Assistive technologies can read
the current values but cannot change them:

- **Web** (`src/lib/color-picker/HsvPicker.svelte`): the SV area and hue strip
  expose `role="slider"`, `tabindex`, and ARIA values, but there are no
  keyboard handlers — Tab reaches the controls, arrow keys do nothing.
- **Apple** (`apple/Dotorixel/Views/HsvPickerView.swift`): the SV square and
  hue strip expose `accessibilityLabel`/`accessibilityValue`, but no
  adjustable or custom actions — VoiceOver users cannot operate them.

Flagged by coderabbitai and cubic-dev-ai on PR #330 (issue 238); deferred
there because the gap is cross-shell and needs shared interaction design.

## Solution (sketch — needs design decision)

The SV square is two-axis, while adjustable actions/arrow keys are one-axis.
Decide one mapping and mirror it on both shells:

- **Option A — split axes**: expose saturation and brightness as separate
  adjustable elements (Apple: two `accessibilityAdjustableAction` elements;
  web: arrow Left/Right = saturation, Up/Down = brightness on the SV area).
  Hue strip is a plain one-axis slider on both.
- **Option B — custom actions** (Apple): named increment/decrement actions
  per dimension via `accessibilityCustomAction`; web still needs Option A's
  keyboard handling.

Whichever mapping is chosen, edits must reuse the existing model paths
(`HsvPickerModel` on Apple, `hsv` state + `emitColor` on web) so behavior
stays in sync with pointer edits.

## Acceptance criteria

- Web: SV area and hue strip are fully keyboard-operable with visible focus,
  ARIA values updating live.
- Apple: VoiceOver can adjust saturation, brightness, and hue; announced
  values update live.
- Foreground color updates identically to pointer edits on both shells.

## Triage decision (2026-07-20)

> *This was generated by AI during triage.*

**Option A — split axes** is chosen. Rationale: the web needs per-axis arrow-key
handling either way, and exposing saturation and brightness as separate
adjustable elements is the standard VoiceOver idiom; Option B's custom actions
are less discoverable and diverge from the web interaction model. Redundancy
check confirmed neither shell has any keyboard/adjustable handling today, and
`accessibilityAdjustableAction` would be the first use of that API family in
the Apple shell. No matching prior rejection (no `.out-of-scope/` KB yet).

## Agent Brief

> *This was generated by AI during triage.*

**Category:** enhancement
**Summary:** Make the HSV picker operable by assistive technologies on both
shells using a split-axis mapping (Option A): saturation and brightness as
independent one-axis adjustments, hue as a plain one-axis slider.

**Current behavior:**
Both shells expose read-only accessibility metadata but no way to change
values without a pointer drag. The web `HsvPicker` component's SV area and hue
strip have `role="slider"`, `tabindex="0"`, and ARIA values but no keyboard
handlers and no visible focus style. The Apple `HsvPickerView`'s SV square and
hue strip have `accessibilityLabel`/`accessibilityValue` but no adjustable or
custom actions, so VoiceOver users cannot operate them.

**Desired behavior:**
- **Web:** With the SV area focused, ArrowLeft/ArrowRight adjust saturation
  and ArrowUp/ArrowDown adjust brightness; with the hue strip focused, arrow
  keys adjust hue. Values clamp at their range ends (no wrap). Handled arrow
  keys call `preventDefault()` so the page does not scroll. Both controls show
  a visible focus ring following the shell's `:focus-visible` convention
  (accent-colored outline via the design-token accent). ARIA attributes
  (`aria-valuenow`/`aria-valuetext`) update live with each keyboard edit.
- **Apple:** The SV square is exposed as two separate adjustable accessibility
  elements (saturation, brightness) and the hue strip as one, each supporting
  VoiceOver increment/decrement, with announced values updating after each
  adjustment.
- **Step sizes:** must allow traversing the full range in a reasonable number
  of operations. Suggested: 1 unit (1 % / 1°) per arrow key on web with a
  larger Shift-modified step, and a coarser step (around 5 % / 10°) for
  VoiceOver adjustments where no modifier exists. Exact values are an
  implementation decision; keep each shell internally consistent.
- **Localization:** any new user-facing or accessibility strings exist in
  en/ko/ja on both shells (web Paraglide messages, Apple String Catalog — the
  catalog completeness test enforces ko/ja).

**Key interfaces:**
- Web `HsvPicker` Svelte component — keyboard edits must reuse the existing
  `hsv` state and `emitColor()` path so behavior stays identical to pointer
  edits (including the last-synced-hex echo guard). The SV area currently
  exposes only saturation via `aria-valuenow`; ensure `aria-valuetext`
  continues to describe both axes after keyboard edits.
- The web shell already has an arrow-key handling exemplar in its tab strip
  component (switch on `event.key` with `preventDefault()`); follow that
  pattern.
- Apple `HsvPickerModel` — `setSaturationValue(s:v:)` and `setHue(_:)` are
  the only edit paths; adjustable actions must call these and then invoke the
  view's `onColorChange` with `model.color`, exactly as the drag gestures do.

**Acceptance criteria:**
- [ ] Web: SV area and hue strip are fully keyboard-operable per the mapping
      above, with clamping at range ends
- [ ] Web: both controls show a visible focus indicator consistent with the
      existing `:focus-visible` convention
- [ ] Web: ARIA values update live on keyboard edits; a component test covers
      the keyboard interactions (none exists for `HsvPicker` today)
- [ ] Apple: VoiceOver can adjust saturation, brightness, and hue as three
      one-axis adjustable elements; announced values update live
- [ ] Apple: adjustable-action edit paths are covered at the model level by
      unit tests (the existing model test suite pattern)
- [ ] Both shells: assistive edits update the foreground color identically to
      pointer edits
- [ ] All new strings localized en/ko/ja on both shells

**Out of scope:**
- Option B (`accessibilityCustomAction`) — rejected by this triage
- XCUITest adoption — VoiceOver end-to-end behavior is not automatable in the
  current test setup; assert at the model level
- Any change to pointer/drag interaction or picker visuals
- Accessibility of other color controls (palette swatches, recent colors)

### Key Decisions
- Hue arrow keys follow the **visual direction** (ArrowDown = indicator moves down = hue increases), chosen over the ARIA "Up = increase" convention; announced values stay correct for screen-reader users either way.
- Step sizes: web 1 % / 1° with Shift ×10; Apple VoiceOver 5 % / 10° (no Shift-like modifier exists). Per-shell internal consistency preferred over cross-shell equality, as the brief allowed.
- Percent accessibility values (`%lld%%`) are symbol-only and marked `shouldTranslate: false` in the String Catalog.

### Notes
- VoiceOver behavior is asserted at the model level only (XCUITest out of scope per the brief); view wiring follows the standard split-element adjustable idiom.
- Web component tests simulate the controlled-component parent echo (`rerender` with the emitted hex) — the picker resets its internal HSV if the parent never echoes the emitted color back, which is pre-existing contract behavior shared with pointer drags.
