## Problem Statement

Drawing on iPad with an Apple Pencil is a priority ergonomic direction for
DOTORIXEL (casual + learning), but today the editor treats the pencil as just
another pointer:

- **Resting a palm on the glass breaks drawing.** A palm registers as one or
  two direct touches. A palm that lands before the pencil blocks the pencil
  from starting a stroke at all (the episode rule admits no begin while any
  touch is down). A palm that produces two contact points fires the Gesture
  Signal and ends an in-flight pencil stroke mid-line. A palm that lands
  alone and drags can paint stray marks the user never intended.
- **There is no way to see where the pencil will land.** Pixel art demands
  per-pixel precision; at moderate zoom the pencil tip covers several canvas
  pixels. On hover-capable hardware (iPad Pro M2+ with Apple Pencil 2 / Pro),
  the pencil reports its position before touching the glass, but the editor
  ignores it — the user only learns which pixel they hit after the paint
  lands.

The natural pencil workflow — hand resting on the tablet, hovering to aim,
touching down to paint — is exactly the workflow the current input routing
punishes.

## Solution

Make the pencil a first-class citizen of the Apple shell's input routing:

- **Palm rejection (all iPads):** the pencil takes priority over direct
  touches. A pencil begin is admitted even while fingers or a palm are down;
  an in-flight pencil stroke ignores direct touches entirely (they neither
  draw, nor fire the Gesture Signal, nor pan/zoom the viewport); and while
  the pencil hovers near the glass, direct touches never begin a stroke.
- **Hover preview (hover-capable iPads):** while the pencil hovers over the
  canvas, the pixel cell it would paint is highlighted, tracking the pencil
  live and disappearing the moment the pencil touches down, leaves hover
  range, or points outside the canvas.

Finger drawing still works exactly as today when no pencil is involved —
the existing Originating Touch, Gesture Signal, Deferred Begin, and episode
rules are unchanged for finger-only interaction.

## User Stories

1. As an iPad artist drawing with an Apple Pencil, I want to rest my palm on the screen while drawing, so that I can draw in a natural hand posture without stray marks.
2. As an iPad artist, I want a palm that lands before my pencil tip to not block the pencil stroke, so that I don't have to lift my hand before every line.
3. As an iPad artist, I want a palm producing two contact points to not end my in-flight pencil stroke, so that long lines don't get cut off mid-stroke.
4. As an iPad artist, I want a lone palm or finger drag that happens while my pencil is hovering to paint nothing, so that aiming with the pencil never produces accidental marks.
5. As an iPad artist mid-pencil-stroke, I want finger touches to neither pan nor zoom the viewport, so that the canvas doesn't slide underneath my line while I draw.
6. As an iPad artist, I want viewport gestures (pinch-zoom, two-finger pan) to resume working normally once my pencil lifts, so that palm rejection never costs me navigation.
7. As a pixel artist on a hover-capable iPad, I want to see which canvas pixel my pencil is pointing at before touching down, so that I can place single pixels precisely.
8. As a pixel artist, I want the hover highlight to track the pencil live as I move it, so that aiming feels immediate.
9. As a pixel artist, I want the hover highlight to disappear the moment I touch down, so that it never obscures the paint I'm laying.
10. As a pixel artist, I want the hover highlight to disappear when the pencil leaves hover range or moves off the canvas, so that no stale indicator lingers.
11. As a pixel artist, I want the hover highlight to stay glued to the correct canvas pixel when I zoom or pan, so that the preview is trustworthy at any viewport state.
12. As a pixel artist using the eraser or any other drawing tool, I want the same target-cell highlight while hovering, so that precision aiming works regardless of tool.
13. As an iPad artist on older hardware without pencil hover, I want palm rejection to still protect my pencil strokes, so that the feature does not depend on the newest iPad.
14. As an iPad artist who draws with fingers (no pencil in hand), I want finger drawing, tap-to-dot, pinch-zoom, and two-finger pan to behave exactly as before, so that palm rejection never regresses touch drawing.
15. As an iPad artist, I want a finger stroke that was already in flight when my pencil touches down to resolve cleanly (drawn pixels commit as one undo entry; an uncommitted tap leaves nothing), so that pencil priority never corrupts history or leaves stray dots.
16. As a user undoing after a pencil session with palm contact, I want each pencil stroke to be exactly one history entry with no phantom palm entries, so that undo remains predictable.
17. As a macOS user, I want mouse and trackpad behavior untouched, so that the pencil work carries no desktop regressions.
18. As a developer, I want the palm-rejection rules pinned by tests at the routing seam (asserting pixel and history outcomes), so that future input work cannot silently regress them.
19. As a developer, I want the hover preview's visibility rules pinned by tests at the editor-state seam, so that the highlight's appear/disappear contract is regression-defended.

## Implementation Decisions

- **Shell-native placement, no Rust core involvement** — same Core Placement
  rationale as issue 245: the logic is inseparable from UIKit touch identity,
  hover recognizers, and gesture-recognizer interplay; it sits on the
  per-event input hot path; the web shell has no equivalent to share.
- **Palm rejection lands in `TouchStrokeRouter`** — the existing pure,
  platform-neutral routing seam established by 245 (explicitly left as "the
  seam palm rejection will refine"). The router stays a generic pure struct
  with no UIKit dependency. New rules, confirmed with the maintainer:
  1. **Pencil begin is always admitted.** Down direct touches no longer block
     a pencil begin. An in-flight direct-finger stroke resolves through the
     same semantics as the Gesture Signal: a dragged stroke commits what it
     drew (one history entry); a stroke still holding only its Deferred Begin
     is discarded (no pixels, no history).
  2. **An active pencil stroke ignores direct touches entirely.** Any number
     of direct touches may begin, move, and end during a pencil stroke
     without drawing, without firing the Gesture Signal, and without ending
     the stroke — extending 245's lone-resting-finger tolerance to the full
     palm. Only the originating pencil touch's move/end/cancel drives the
     stroke.
  3. **Pencil hover blocks direct-touch stroke begins.** The router gains a
     pencil-hover input (a boolean state change fed by the view). While the
     pencil hovers, a direct touch never begins a stroke (its Deferred Begin
     is never created); viewport gestures remain available. Hover state does
     not end a finger stroke already in flight — it only gates new begins.
  4. The episode rule ("no stroke until every touch lifts") and all
     finger-only behavior are unchanged.
  - On hardware without pencil hover, rule 3 simply never triggers; rules 1
    and 2 carry the palm rejection.
- **Viewport gestures are suppressed during a pencil stroke.** The pinch and
  two-finger-pan recognizers must not operate while a pencil stroke is
  active (rule 2's "ignore" includes navigation). Gate the recognizers'
  begin on the stroke state; they resume once the pencil lifts.
- **Hover preview state lives on the editor state** (the `@Observable`
  editor state object) as a published **Hover Point** — the pencil's current
  target in canvas coordinates, or nil when no preview should show. The
  coordinator feeds it from the view's hover events using the existing
  screen→canvas conversion; SwiftUI renders the highlight as an overlay
  above the Metal view (the Loupe's established pattern), deriving the
  cell's display rect from the viewport transform so pan/zoom re-position it
  for free.
- **Visibility contract of the Hover Point:** non-nil only while the pencil
  is in hover range over a canvas pixel (in-bounds after screen→canvas
  conversion); cleared when the pencil exits hover range, moves off-canvas,
  or a stroke begins. Tool-independent: every drawing tool gets the same
  single-cell target highlight.
- **Hover input arrives via a hover gesture recognizer restricted to the
  pencil touch type** on the iPad input view. Trackpad/mouse pointer hover
  is deliberately not admitted (confirmed) — the recognizer restriction is
  the only thing to relax if that extension is wanted later. The same hover
  events feed both the Hover Point and the router's rule-3 hover state.
  Hover z-offset, azimuth, and altitude are ignored.
- **No settings toggle.** Palm rejection is automatic; no "finger draws"
  preference is introduced (revisit only if the automatic rules prove
  insufficient in real use).
- **macOS paths are untouched**; the `CanvasInputDelegate` drawing contract
  is unchanged (hover feeds state, not drawing commands — extend the
  delegate only if routing genuinely cannot stay in the view layer).
- **Loupe input-source mapping is unchanged** (pencil continues to use the
  mouse-style offsets).
- **Domain vocabulary**: register **Hover Point** and the pencil-priority
  rules (as an extension of Gesture Signal / Originating Touch entries) in
  `CONTEXT.md` during implementation, as 245 did for the input vocabulary.

## Testing Decisions

- **Test external behavior, not implementation details**: assert
  stroke-lifecycle outcomes — canvas pixels, history entries, session
  state, Hover Point values — never event-forwarding call sequences (the
  lesson pinned by issue 248 and already followed by the 245 tests).
- **Router rules** are tested in `TouchStrokeRouter`'s existing Swift
  Testing suite via the `RoutedEditor` harness (drives router commands into
  a real editor state and asserts pixels + history), extended with
  pencil-vs-finger scenarios: palm-then-pencil begin, palm landing
  mid-pencil-stroke (one and two touches), hover-blocked finger begin,
  finger stroke resolution on pencil preemption, post-episode recovery.
- **Hover Point visibility rules** are tested at the editor-state seam:
  feeding hover enter/move/exit and stroke begins, asserting the published
  Hover Point (position, nil-ness) — no rendering assertions.
- **Prior art**: `TouchStrokeRouterTests` (router harness style),
  the editor-state suites under `apple/DotorixelTests/` (Swift Testing,
  pinned iPad simulator per `apple/DotorixelTests/README.md`).
- **Manual device pass required**: the simulator cannot simulate pencil
  hover, real palm contact geometry, or recognizer interplay. Keep the
  recognizer/view wiring thin so everything meaningful is unit-tested; note
  that 245 already flagged a manual pass of pinch/pan interplay as
  outstanding — fold it into this feature's device verification.

## Out of Scope

- **Trackpad / mouse pointer hover preview** (iPad or macOS) — the seam
  (Hover Point + overlay) is input-agnostic, but only pencil hover is
  admitted now.
- **Web shell** — no changes; this feature is Apple-specific, beyond web
  parity by design.
- **Pencil dynamics** — pressure, tilt, azimuth, barrel roll, and
  hover z-offset (e.g. preview scaling with distance) are not read.
- **Pencil hardware gestures** — double-tap tool switch, Pencil Pro
  squeeze; separate features if ever wanted.
- **A "finger drawing" settings toggle** (Procreate-style touch-action
  preference) — automatic rules only, per maintainer decision.
- **Hover-driven Loupe or eyedropper preview** — the Loupe remains
  stroke-driven.
- **Reworking finger-only routing** — Deferred Begin, Gesture Signal, and
  episode semantics for finger-only interaction stay exactly as 245 built
  them.

## Further Notes

- Pencil hover requires iPad Pro (M2 or later) with Apple Pencil 2 / Pencil
  Pro; palm rejection rules 1–2 work on every pencil-capable iPad. iPadOS
  already filters most palm touches system-side when a pencil is in use —
  these rules are the editor-level backstop for the contacts that get
  through.
- Hover events fire at high frequency; the Hover Point update path should
  stay allocation-light, but the Loupe already survives per-move SwiftUI
  updates during strokes, so no special machinery is expected.
- The router's hover input is a plain boolean state change, keeping the
  router free of UIKit types and trivially drivable from tests.
- 245's `syncDownTouches` begin-boundary reconciliation already excludes
  hover phases from the down-touch snapshot, so a hovering pencil never
  blocks finger drawing through *that* path — rule 3 is the deliberate,
  opposite-direction gate.
