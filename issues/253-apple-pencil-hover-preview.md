## What to build

While the Apple Pencil hovers over the canvas (iPad Pro M2+ with Pencil 2 /
Pencil Pro), highlight the canvas pixel cell it would paint, live.

- A hover gesture recognizer restricted to the pencil touch type on the
  iPad input view feeds hover enter/move/exit through the coordinator,
  which converts to canvas coordinates with the existing screen→canvas
  conversion.
- The editor state publishes a **Hover Point** — the pencil's current
  target in canvas coordinates, or nil when no preview should show.
- A SwiftUI overlay above the Metal view (the Loupe's established pattern)
  renders the highlighted cell, deriving its display rect from the viewport
  transform so pan/zoom re-position it for free.
- Visibility contract: non-nil only while the pencil is in hover range over
  an in-bounds canvas pixel; cleared when the pencil exits hover range,
  moves off-canvas, or a stroke begins. Tool-independent — every drawing
  tool gets the same single-cell target highlight.
- Hover z-offset, azimuth, and altitude are ignored. Trackpad / mouse
  pointer hover is not admitted (the recognizer's touch-type restriction is
  the only thing to relax later).
- Register **Hover Point** in the domain vocabulary.

## Acceptance criteria

- Pencil hover over the canvas publishes a Hover Point at the correct
  canvas cell, and the overlay highlights exactly that cell.
- Moving the hovering pencil updates the highlight live; panning or zooming
  the viewport keeps the highlight glued to the same canvas cell.
- The Hover Point clears (and the highlight disappears) when the pencil
  leaves hover range, when the converted position falls outside the canvas
  bounds, and the moment a stroke begins.
- Finger and indirect-pointer input never produce a Hover Point.
- The visibility contract is pinned by Swift Testing tests at the
  editor-state seam — feeding hover enter/move/exit and stroke begins,
  asserting the published Hover Point (position, nil-ness) — with no
  rendering assertions.
- The hover update path adds no per-event allocations beyond what the
  Loupe's pointer pushes already establish.
- macOS target builds clean; macOS input paths untouched.

### Key Decisions

- **Hover feeds state, not a drawing command**, so the `CanvasInputDelegate`
  protocol is untouched — `handleHover` sits on the Coordinator alongside
  `handlePinch`/`handleTwoFingerPan`, reading `editorState`/`viewport`
  directly (PRD: extend the delegate only if routing can't stay in the view).
- **The Hover Point is a `ScreenCanvasCoords`** (the existing canvas-cell
  type), keeping domain vocabulary consistent with `beginStroke`.
- **Bounds check lives on `EditorState`** (`isInCanvasBounds`), so the
  "in-bounds after conversion" half of the visibility contract is unit-tested
  at the state seam rather than in the untestable view layer.
- **The hover recognizer is excluded from `viewportGestureRecognizers`**, so
  252's `gestureRecognizerShouldBegin` gate leaves it alone with no exemption
  code needed (the gate already returns `true` for non-viewport recognizers).
