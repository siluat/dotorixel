## What to build

Complete the palm rejection story on hover-capable hardware (PRD rule 3):
while the pencil hovers near the glass, a palm or finger that lands never
starts a stroke — aiming with the pencil can never produce accidental
marks.

- The routing seam gains a pencil-hover input — a plain boolean state
  change fed by the view from the same hover events issue 253 wires up —
  keeping the seam pure and UIKit-free.
- While the pencil hovers, a direct touch never begins a stroke (its
  Deferred Begin is never created). Viewport gestures (pinch-zoom,
  two-finger pan) remain available.
- Hover state only gates *new* begins: it does not end a finger stroke
  already in flight, and it never affects pencil begins.
- On hardware without pencil hover the input simply never fires; behavior
  there is fully covered by issue 252's contact rules.

## Acceptance criteria

- While the pencil hovers: a single direct touch that lands, drags, and
  lifts paints nothing and leaves no history entry.
- After the pencil exits hover range (and all touches lift), a fresh single
  finger draws normally again.
- The pencil entering hover range while a finger stroke is in flight does
  not end or alter that stroke.
- Two direct touches while the pencil hovers still pinch-zoom and
  two-finger-pan normally.
- A pencil begin during hover is unaffected by the gate (hover → touch-down
  draws immediately).
- New behavior is pinned by Swift Testing tests at the routing seam via the
  existing routed-editor harness, asserting pixels + history outcomes.
- The routing seam remains free of UIKit types.

### Key Decisions

- **Gate placement — after the pencil branch, before the stroke switch.** A
  pencil begin is never gated (criterion 5), and a hovering pencil makes direct
  touches fully inert to the stroke machine (never starts, discards, or ends a
  stroke) while still being censused for episode-blocking — the cleanest
  palm-rejection semantic.
- **Plain `Bool` input, reusing 253's hover recognizer.** The routing seam stays
  UIKit-free; the view feeds `setPencilHovering` from the same hover events that
  drive the Hover Point.
- **Native-per-shell placement, not Rust core.** Input routing is shell-owned
  (CONTEXT.md); the web counterpart ("Web pen priority") is separate backlog work
  on the canvas interaction machine.
